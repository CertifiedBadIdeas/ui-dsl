# Kraft UI DSL

Kraft UI DSL — это Kotlin-библиотека для описания, проверки, оптимизации, предпросмотра и генерации интерфейсов для модов Minecraft и других JVM-хостов.

Основной путь рассчитан на этап сборки:

```text
Kotlin-описание интерфейса -> ScreenProgram -> диагностика -> оптимизация -> предпросмотр -> сгенерированный код/ресурсы цели
```

Библиотека пока активно развивается. Её публичный API, проходы компилятора, вид сгенерированного кода и возможности стилизации ещё могут меняться, пока архитектура окончательно закрепляется.

## Цели

- Иметь один источник истины для экрана, а не дублировать разметку между игровым кодом, предпросмотром и сгенерированным кодом.
- Ловить ошибки разметки, текста, неподдержанных примитивов и несовместимости с целью вывода до запуска Minecraft.
- Генерировать код и ресурсы под конкретную цель вывода, а не тащить тяжёлый исполнитель интерфейса туда, где нужны только готовые команды рисования.
- Делать предпросмотр достаточно близким к игре, чтобы визуальные ошибки находились в сборочном процессе.
- Описывать действия типизированно и явно, без строковых команд и непрозрачных лямбд с захваченным состоянием.

## Архитектура

- `foundation` содержит элементы интерфейса, значения, цвета, модификаторы, действия и базовые понятия рисования.
- `program` компилирует дерево интерфейса в `ScreenProgram`, анализирует его, оптимизирует, сравнивает цели вывода и генерирует код.
- `preview` рисует программы в изображения для разработки и отчётов сборки.
- Стили описываются данными, включая процедурные текстуры: клетчатые заливки, латунные рамки и похожие примитивы.
- Целевые модули решают, как проверенная и оптимизированная программа превращается в код хоста, например в вызовы Minecraft `GuiGraphics`.

Исполнитель программы остаётся эталонным путём для проверки, предпросмотра и сравнения целей вывода. Для серьёзных экранов Minecraft он не должен быть основным способом поставки интерфейса.

## Модель Использования

1. Описать интерфейс на Kotlin через DSL и типизированные модели состояния/действий.
2. Скомпилировать описание в `ScreenProgram`.
3. Запустить диагностику текста, разметки, областей нажатия, неподдержанных возможностей цели и неверных размеров.
4. Оптимизировать программу с явными настройками компилятора.
5. Отрисовать предпросмотр и сравнить целевой результат с эталонным путём.
6. Сгенерировать код и ресурсы для проекта-хоста.
7. Обработать типизированные действия во внешнем коде хоста.

Интерфейс описывает, что пользователь может сделать. Изменение предметной области, игровое состояние, сеть и серверная истина остаются вне описания интерфейса.

## Текущий Статус API

Точный вид DSL ещё меняется. Пока публичный API не объявлен стабильным, рабочие примеры лучше смотреть в текущих тестах и проектах, которые используют библиотеку.

Направление уже закреплено: описание интерфейса должно использовать типизированное состояние, типизированные действия, явные модификаторы, явные настройки компилятора и сгенерированные цели вывода, которым не нужно обходить исходное дерево интерфейса во время работы.

## Оптимизация

Оптимизация включается явно и даёт отчёт. Компилятор может сворачивать константы, удалять мёртвые ветки, заранее считать области нажатия, кэшировать разметку текста, группировать блоки видимости и запекать доказанно статичное рисование в текстуры.

Каждая оптимизация должна объясняться в отчёте:

```text
StaticTextureBaking:
  applied: /root/window, 58 ops -> 1 texture, 232x140
  skipped: /root/mixture/list, contains dynamic text
```

Оптимизация не должна менять поведение. Цели вывода сравниваются через снимки и события ввода, чтобы ловить расхождения между эталонным исполнителем, предпросмотром и сгенерированным кодом.

## Стили

Стили — это данные, а не вручную написанная логика рисования. Сейчас есть:

- сплошные цвета и поверхности;
- слоистые текстуры;
- клетчатые заливки;
- девятичастные и сегментные текстурные области;
- процедурные латунные рамки;
- запекание сложных доказанно статичных участков в текстуры.

Целевое направление — широкая система стилей, которую можно переиспользовать между модами и при этом компилировать в небольшой код под конкретную цель вывода.

## Подключение Из Исходников

Библиотека рассчитана на подключение из исходников через Git-подмодуль или составную сборку Gradle:

```bash
git submodule add https://github.com/CertifiedBadIdeas/ui-dsl vendor/ui-dsl
```

```kotlin
includeBuild("vendor/ui-dsl")
```

Затем модуль можно подключить в проекте-потребителе:

```kotlin
dependencies {
    implementation("ru.lazyhat:kraft-ui-dsl")
}
```

Для сгенерированных экранов Minecraft итоговый мод должен зависеть от сгенерированного кода и ресурсов, а не от большого исполнителя интерфейса, если этот исполнитель не нужен намеренно.

## Проверка

```bash
./gradlew test
```

Проектам-потребителям также стоит запускать генерацию предпросмотра и сравнение целей вывода как часть обычных проверок.
